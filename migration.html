Migration Strategy

Phased migration approach for moving a legacy operational fulfilment platform from on-premises infrastructure into an AWS-first architecture while protecting operational continuity, reducing risk, and enabling future modernisation.

Migration Objective

The migration objective is to safely transition critical fulfilment platform capabilities to AWS using a phased, risk-managed approach. The strategy prioritises business continuity, rollback readiness, dependency understanding, validation, and operational support readiness rather than a high-risk “big bang” migration.

Protect Operations

Maintain fulfilment continuity and avoid unnecessary disruption to order processing, inventory allocation and robotic workflows.

Reduce Migration Risk

Use controlled migration waves, testing, validation and rollback planning to reduce operational risk.

Improve Resilience

Introduce AWS services that support improved availability, recovery and operational stability.

Enable Modernisation

Create the foundation for refactoring, CI/CD, observability, automation and AI-assisted operations.

Strengthen Governance

Introduce migration controls, stakeholder checkpoints, security reviews and architecture decision tracking.

Validate Before Cutover

Confirm performance, connectivity, recovery, monitoring and support readiness before moving production workloads.

Migration Waves

Workloads are grouped into migration waves based on business criticality, dependency complexity, operational risk and readiness.

Wave 0: Foundation

Set up AWS account structure, IAM, networking, security baselines, monitoring foundations and backup policies.

Wave 1: Supporting Services

Migrate low-risk supporting services such as reporting exports, documentation stores, logs and operational artefacts.

Wave 2: Integration Services

Introduce AWS messaging patterns, test service communication and validate event-driven integration behaviour.

Wave 3: Application Workloads

Migrate fulfilment application services using EC2 or ECS depending on workload readiness and refactoring complexity.

Wave 4: Database Platform

Migrate SQL workloads to Amazon RDS SQL Server with backup, restore, performance and connectivity validation.

Wave 5: Operational Cutover

Perform controlled production cutover with rollback readiness, monitoring, support coverage and post-cutover stabilisation.

Migration Strategy Decisions

Rehost Where Necessary

Some workloads may initially be rehosted to EC2 to reduce migration risk and avoid forcing premature refactoring. This supports operational continuity while preparing for later modernisation.

Refactor Selectively

Components with clear scalability, reliability or deployment benefits may be refactored earlier into containerised services, APIs or event-driven workflows.

Use Managed Services

Managed services such as RDS, S3, CloudWatch, Secrets Manager, AWS Backup, SQS and SNS reduce operational overhead and improve resilience.

Design for Rollback

Every migration wave should include rollback criteria, validation checkpoints, decision gates and stakeholder sign-off.

Cutover & Rollback Planning

Cutover Window

Schedule cutover during a controlled operational window with agreed business stakeholders and support coverage.

Pre-Cutover Validation

Validate connectivity, application behaviour, database access, messaging, monitoring, backups and user access.

Rollback Criteria

Define clear rollback triggers such as failed transactions, unacceptable latency, data sync issues or critical service failure.

Rollback Procedure

Document how traffic, services, database connectivity and operational processes will be restored if rollback is required.

Post-Cutover Monitoring

Increase monitoring intensity after cutover to detect performance degradation, errors, queue delays and user impact quickly.

Support Readiness

Ensure support teams understand the new environment, escalation paths, runbooks, access model and operational dashboards.

Migration Risks & Mitigations

Dependency Gaps

Risk: Unknown integrations cause failure after migration.
Mitigation: Dependency mapping, testing and phased cutover.

Performance Regression

Risk: Cloud-hosted services perform worse than current workloads.
Mitigation: Load testing, sizing, monitoring and performance baselines.

Data Migration Issues

Risk: Incomplete or inconsistent data migration.
Mitigation: Backup, restore validation, reconciliation and rollback planning.

Security Misconfiguration

Risk: Poor IAM, network or secrets configuration creates exposure.
Mitigation: Least privilege, security groups, secrets management and reviews.

Operational Readiness Gap

Risk: Support teams are not ready to operate the new platform.
Mitigation: Runbooks, dashboards, training and hypercare support.

Change Fatigue

Risk: Too much change increases support pressure.
Mitigation: Migration waves, communication plan and phased modernisation.

Migration Readiness Checklist

Architecture Approved

Target architecture, security model, network design and migration approach reviewed and agreed.

Landing Zone Ready

AWS account, IAM, VPC, security groups, monitoring and backup baselines prepared.

Dependencies Mapped

Application, database, messaging, reporting, print and robotic system dependencies documented.

Testing Complete

Connectivity, functional, performance, recovery and monitoring validation completed.

Rollback Ready

Rollback criteria, ownership, decision points and restoration steps documented.

Support Ready

Runbooks, dashboards, alerting, access and escalation processes prepared for production support.

Improved Operational Visibility

Foundation for Modernisation

Create the platform foundation for CI/CD, observability, automation and AI-assisted support.

Improved Security Baseline

Introduce IAM controls, secrets management, auditability and network segmentation.

Business Continuity Improvement

Reduce downtime risk and improve resilience for fulfilment operations and customer delivery workflows.
