Phase 2: Migrate

AWS Migration Strategy

A phased migration approach for moving Apex Operational Systems from a legacy fulfilment platform to a resilient AWS target-state architecture with reduced operational risk and controlled cutover.

← Back to Portfolio View Case Study Target Architecture

Migration Objective

The migration approach focuses on moving the fulfilment platform to AWS without increasing operational risk. The strategy prioritises resilience, observability, rollback capability, security and business continuity.

Business Continuity

Protect fulfilment operations during migration by using phased waves, validation checkpoints and rollback planning.

Risk Reduction

Move high-risk components gradually, starting with observability, backup and supporting services before core workload migration.

Modernisation Foundation

Prepare the platform for containerisation, managed databases, managed messaging and AI-assisted operations.

Migration Principles

Phased Migration

Use migration waves instead of a single big-bang cutover to reduce delivery and operational risk.

Rollback First

Each migration stage should have a defined rollback path before production cutover is approved.

Observe Before Migrating

Monitoring, logging and alerting should be introduced early so migration decisions are based on operational evidence.

Security by Design

IAM, encryption, secrets management, audit logging and governance controls are built into the migration foundation.

Managed Services First

Prefer AWS managed services where they reduce patching, failover, backup and operational support overhead.

Business Validation

Every migrated component must be validated against fulfilment workflows, reporting, robot task flow and support procedures.

Landing Zone Approach

Before migrating workloads, the AWS foundation must provide secure networking, access control, logging, monitoring and governance.

Network Foundation

VPC, public/private subnets, routing, NAT gateways, security groups and controlled ingress through Application Load Balancer.

Identity & Access

IAM roles, least privilege permissions, MFA, service roles and controlled access to production resources.

Security Baseline

KMS encryption, Secrets Manager, CloudTrail, GuardDuty, Security Hub and AWS Config for governance visibility.

Operational Baseline

CloudWatch Logs, alarms, dashboards, SNS notifications and OpenSearch for searchable operational data.

Backup Baseline

AWS Backup policies for database and knowledge artefacts, with restore testing before production cutover.

Automation Baseline

Systems Manager runbooks and patch management to reduce manual recovery and operational intervention.

Migration Waves

Migration is delivered in controlled waves so that operational visibility, recovery capability and platform stability improve before core workload cutover.

Wave 1: Foundation & Observability

Set up VPC, IAM, CloudTrail, CloudWatch, OpenSearch, SNS, dashboards, backup policies and security baseline.

Wave 2: Messaging & Integration

Introduce Amazon MQ, EventBridge and Dead Letter Queue patterns to improve messaging reliability and failure handling.

Wave 3: Application Services

Containerise fulfilment services onto ECS Fargate across multiple Availability Zones behind an Application Load Balancer.

Wave 4: Data Migration

Migrate SQL workloads to Amazon RDS for SQL Server Multi-AZ, introduce reporting replica and validate backup/restore.

Wave 5: Operational Cutover

Complete production cutover using test evidence, support readiness, rollback procedures and business sign-off.

Wave 6: AI Operations Enablement

Load runbooks, tickets, known issues and architecture documents into the AI Operations Knowledge Base.

Cutover Strategy

Before Cutover

  • Confirm backups and restore testing.
  • Validate ECS services, RDS, MQ and monitoring.
  • Complete performance and integration testing.
  • Prepare rollback plan and support rota.

During Cutover

  • Freeze non-essential changes.
  • Redirect traffic through Route 53 and ALB.
  • Monitor CloudWatch dashboards and alerts.
  • Validate fulfilment, messaging and robot task flow.

After Cutover

  • Monitor queue depth, database performance and application logs.
  • Confirm reporting and operational visibility.
  • Review incidents and update runbooks.
  • Capture lessons learned for future waves.

Rollback Conditions

  • Critical fulfilment workflow failure.
  • Unresolved database performance issue.
  • Messaging backlog exceeding agreed threshold.
  • Robot task execution failure impacting operations.

Migration Risks & Mitigations

Risk Impact Mitigation
Production downtime during cutover Fulfilment disruption and delayed orders Phased cutover, rollback plan, test windows and business validation checkpoints.
SQL performance degradation Slow transactions, reporting delays and operational instability Performance testing, RDS sizing, reporting replica and CloudWatch monitoring.
Messaging backlog during migration Delayed robot tasks and system-to-system communication issues Amazon MQ validation, queue depth alarms, DLQ handling and controlled traffic ramp-up.
Knowledge gap across support teams Longer incident investigations and inconsistent recovery Runbooks, AI Operations Knowledge Base, support readiness sessions and post-cutover hypercare.
Security misconfiguration Unauthorized access, audit gaps or compliance concerns IAM least privilege, CloudTrail, GuardDuty, Security Hub, AWS Config and pre-cutover review.

Success Criteria

Platform Availability

Core fulfilment services remain available across AZ failure scenarios and recover within agreed operational targets.

Operational Visibility

Support teams can view logs, alarms, dashboards, queue depth and database performance from centralised tools.

Reduced Manual Recovery

Runbooks, automation and managed services reduce dependency on manual restart, failover and recovery activities.

Validated Data Migration

Operational database, reporting replica and backup/restore processes are validated before production cutover.

Improved Messaging Resilience

Amazon MQ and DLQ handling reduce risk from queue backlogs and failed operational messages.

AI-Assisted Support Ready

Known issues, runbooks and support documents are available through the AI Operations Knowledge Base.

The migration strategy is designed to show how an architect moves from assessment to execution: understand the current state, define the target state, prepare the landing zone, migrate in waves, validate business outcomes and continuously improve operations.
