Target AWS Architecture

AWS-First Intelligent Operations Platform

Target-state architecture for modernising a legacy fulfilment platform into a resilient, observable, secure, automated and AI-assisted operating model on AWS.

Target Architecture Diagram

The target state replaces legacy application servers, unmanaged operational dependencies and reactive support processes with managed AWS services, Multi-AZ resilience, centralised observability, automated operations and AI-assisted support.

This design is not a simple lift-and-shift migration. It modernises the operating model by moving from server-based application hosting, manual monitoring and reactive troubleshooting to containerised services, managed data platforms, proactive observability, automation and AI-assisted operations.

Architecture Vision

The target architecture introduces an AWS-first platform designed to reduce operational risk, improve service resilience, strengthen disaster recovery, increase observability, secure operational access, and create the foundation for AI-assisted troubleshooting and incident response.

AWS Network Foundation

Route 53, VPC, public/private subnets, Application Load Balancer, NAT gateways and Multi-AZ network segmentation.

Application Platform

ECS Fargate replaces legacy APP-1 / APP-2 servers with containerised services running across multiple Availability Zones.

Data & Recovery

Amazon RDS for SQL Server, reporting replica and AWS Backup improve database availability, recovery and reporting separation.

Messaging & Integration

Amazon MQ, EventBridge and Dead Letter Queue improve asynchronous communication, failure handling and operational event routing.

Observability & Operations

CloudWatch, OpenSearch, SNS and Systems Manager provide logging, alerting, dashboards, automation and patch management.

AI Operations Intelligence

Amazon Bedrock Knowledge Base, S3 and OpenSearch vector indexing support AI-assisted troubleshooting and support guidance.

Key AWS Service Decisions

Each AWS service is selected to address a specific operational or architectural weakness from the current environment.

Route 53 & Application Load Balancer

Provide controlled traffic entry, health-checked routing and distribution across ECS services in multiple Availability Zones.

Amazon ECS Fargate

Removes application server management overhead while supporting containerised fulfilment services, scaling and resilience.

Amazon RDS SQL Server

Provides managed SQL Server hosting, Multi-AZ failover, backup integration and improved database operational control.

Amazon MQ

Modernises RabbitMQ-style messaging with managed broker capability, high availability and reduced operational overhead.

CloudWatch & OpenSearch

Centralise logs, metrics, dashboards and searchable operational data to improve investigation and incident response.

Amazon Bedrock Knowledge Base

Uses runbooks, tickets, known issues and architecture documentation to support AI-assisted troubleshooting and knowledge retrieval.

Secrets Manager & KMS

Reduce shared credential risk, improve secrets handling and encrypt sensitive configuration and operational data.

CloudTrail, GuardDuty & Security Hub

Improve auditability, threat detection and security posture visibility across the AWS environment.

AWS Config

Supports configuration tracking, governance, compliance visibility and architecture accountability.

How the Target State Addresses Current Risks

SQL Performance Bottlenecks

RDS SQL Server Multi-AZ, reporting replica and performance monitoring reduce operational database pressure.

Messaging Queue Backlogs

Amazon MQ, EventBridge and Dead Letter Queue improve message resilience, failure handling and integration visibility.

Manual Failover & Recovery

Multi-AZ managed services, AWS Backup and Systems Manager runbooks reduce reliance on manual recovery processes.

No Proactive Monitoring

CloudWatch alarms, dashboards, logs, SNS notifications and OpenSearch improve detection and investigation speed.

Manual Patching

Systems Manager Patch Manager supports controlled patching and operational automation across managed workloads.

Knowledge Silos

Bedrock Knowledge Base and S3 Knowledge Store turn runbooks, known issues and incidents into reusable operational intelligence.

Architecture Trade-Offs

ECS Fargate vs EKS

ECS Fargate is selected to reduce operational complexity while still supporting containerised workloads, scaling and resilience. EKS may be considered later if Kubernetes portability or advanced orchestration becomes a stronger requirement.

Amazon MQ vs Fully Refactored Eventing

Amazon MQ provides a realistic migration path from RabbitMQ-style messaging. EventBridge and DLQ patterns can then support gradual evolution toward more event-driven operations.

Managed Database Approach

RDS SQL Server reduces operational overhead but still requires sizing, licensing review, performance monitoring, backup policy and migration planning.

One Region, Multi-AZ First

The initial target uses eu-west-2 with multiple Availability Zones for practical resilience. Cross-region disaster recovery can be added later if RTO/RPO requirements justify the additional cost and complexity.

AI Operations Capability

The AI operations layer is designed to extend observability and support workflows by helping engineers investigate issues faster, retrieve operational knowledge and follow structured remediation guidance.

Knowledge Retrieval

Search runbooks, tickets, known issues, SOPs, architecture documents and support notes from the S3 Knowledge Store.

Operational Context

Use OpenSearch and vector indexing to connect logs, historic incidents and known patterns to troubleshooting workflows.

Guided Troubleshooting

AI Operations Assistant suggests likely causes, investigation steps, escalation paths and validation checks.

Continuous Learning

Support teams can update operational knowledge after incidents, improving future recommendations and reducing repeated investigations.

Reduced MTTR

AI-assisted support reduces time spent searching documents, tickets and logs during high-pressure operational incidents.

Expected Business Outcomes

Improved Availability

Multi-AZ design and managed services reduce the impact of infrastructure and service failures.

Reduced Operational Overhead

Fargate, RDS, Amazon MQ and Systems Manager reduce manual server management and recovery workload.

Faster Incident Response

CloudWatch, OpenSearch, SNS and AI-assisted troubleshooting improve detection, triage and remediation.

Stronger Security Posture

IAM, KMS, Secrets Manager, CloudTrail, GuardDuty, Security Hub and AWS Config improve control and auditability.

Better Release Readiness

Containerised services create a stronger foundation for future CI/CD, automated deployment and safer rollback patterns.

AI-Assisted Operations

Operational knowledge becomes searchable, reusable and available to engineers during live investigations.
